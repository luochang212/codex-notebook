你团队来了个新人。你想教他"我们团队写 React 组件的规矩"：先写 interface，再写组件，最后写测试；命名用 PascalCase；每个组件必须有 storybook。

你会怎么做？写一个 ESLint 插件强制执行？写一个 CLI 工具自动生成模板？还是……写一份文档，发给他，让他读？

大概率是后者。因为规矩是知识，不是能力。新人不是"做不到"，是"不知道"。你告诉他，他就会了。不需要运行时，不需要 API，不需要依赖管理。

Codex 的 skill 就是这个逻辑。skill 就是一个 `SKILL.md` 文件。没有运行时、没有 API、没有依赖。它的全部"执行"就是被读取后注入模型的上下文。skill 不是代码扩展，而是**文档扩展**。

## 渐进式披露：不用的不占地方

但有一个问题：如果你有 50 个 skill，每个 2000 token，全部注入就是 100K token——整个 context window 被 skill 吃光了。

Codex 的解法是三层渐进式披露：

**第一层：目录。** 每个 turn 开始时，系统把所有可用 skill 的 name + description 渲染成一段目录文本，放进 developer context。模型看到的是"有哪些 skill 可用"，不是 skill 的完整内容。目录的 token 预算是上下文窗口的 2%。128K 窗口，目录最多占 2560 token。

**第二层：正文。** 只有当 skill 被选中（用户显式 mention 或模型自主判断）后，完整的 `SKILL.md` 才被读取并注入。

**第三层：引用。** `SKILL.md` 里可能引用 `references/`、`scripts/`、`assets/`。模型按文档中的路由说明，用工具按需读取。

不用的 skill 只占目录里的一行名字。用了的才占完整 token。50 个 skill 里只有 2 个被用到，那只有 2 个的完整内容进入上下文。

## 谁来决定用哪个 skill？

Codex 把"语义匹配"留给模型，而不是在 core 里实现一个 skill 检索系统。判断规则直接写进 developer instructions：

> 如果用户指名了一个 skill（用 `$SkillName` 或纯文本），或者任务明显匹配某个 skill 的描述，你必须使用该 skill。

用户显式 mention 时，core 会在模型采样前解析并注入。命名冲突处理很保守：如果一个名字同时匹配多个 skill，宁可不选也不注入错误的 skill。

## Skill 的更新

本地环境有 file watcher，约 10 秒节流后清空 skill cache 并通知客户端。但有一个重要边界：**当前 active turn 持有的 snapshot 不会自动突变。** 变化影响的是后续 turn 的 snapshot 构建。你不会在 turn 中间突然看到 skill 消失或变化。

## 为什么不用插件代码？

因为 skill 解决的是"教模型做事的方法"，不是"给模型新的能力"。

代码风格、工作流程、领域知识、团队规范——这些是知识。知识用文档传递就够了。你不需要一个运行时来"执行"代码风格。你只需要告诉模型"我们这样写代码"，它就会这样写。

插件代码能做什么 skill 做不到的？调 API、改系统状态、执行代码。但这些是"能力"，不是"知识"。能力需要运行时，需要 MCP（下一章）。知识不需要。

而且插件代码有代价：需要代码审查、沙箱执行、依赖管理、版本锁定。一个 bug 可能导致系统崩溃。Markdown skill 的最坏情况是什么？模型行为不佳。不会崩溃，不会泄漏，不会破坏数据。人类直接阅读就能审计。

## 代价

skill 没有运行时能力。它不能调 API、不能修改系统状态、不能执行代码。它的有效性完全取决于模型是否能正确理解和遵循文档中的指令。

语义匹配不精确。模型可能忽略应该使用的 skill，或错误使用不相关的 skill。没有硬保证——skill 的"规则"只是 prompt 文本，不是硬约束。

但对于"教模型做事的方法"这个需求，markdown 是最低成本、最安全、最可审计的方案。创建门槛是会写文档就行，不需要会编程。

skill 把"扩展 agent"从"写代码"降格为"写文档"。这不是能力的扩展，而是知识的注入。
