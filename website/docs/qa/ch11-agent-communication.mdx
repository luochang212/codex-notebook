你给三个同事分别布置了一个任务。然后你继续干自己的事。某个同事做完了，他不会拍你肩膀说"嘿，我做完了"——他会在你的收件箱里放一张备忘录。你下次查看收件箱时看到它，然后决定下一步。

Codex 的多 agent 通信就是这个模型。子 agent 完成后不会"回调"父 agent，不会中断父 agent 的当前工作。它只是往父 agent 的 input queue 里塞了一条消息。父 agent 下次空闲时看到它，就像看到一条普通的用户输入一样处理。

没有 RPC，没有服务发现，没有消息队列。多 agent 通信被还原成了单 agent 的输入处理。

## V1：一个监听者在走廊里等

V1 协议下，spawn 子 agent 时系统会启动一个 detached 的 completion watcher。它不做任何事，只是订阅子 agent 的状态 channel，然后等。

等什么呢？等子 agent 的状态变成 final——Completed、Errored、或 Interrupted。一旦状态变为 final，watcher 构造一条通知消息（包含子 agent 的引用名和最终状态），然后调用 `inject_user_message_without_turn` 把消息塞进父 agent 的 input queue。

关键：这条消息不会立即触发父 agent 的新 turn。它只是排队。父 agent 可能正在忙别的事，这条消息就安静地等着。等父 agent 下次空闲，它会被当作一条普通的用户输入处理。

watcher 是一次性的。通知发完，它就消失了。如果 watcher 在子 agent 完成前崩溃了，通知丢失。没有重试机制。

## V2：子 agent 自己发备忘录

V2 协议下，没有外部 watcher。子 agent 的 Session 在自己的 terminal turn 事件处理中主动通知父 agent。

区别在哪？V1 是"别人帮你盯着，做完了别人通知你"。V2 是"我自己做完了，我自己发备忘录"。

V2 的通知走 `InterAgentCommunication`——一个结构化的 agent 间通信对象，包含发送者路径、接收者路径、消息内容和类型。不是纯文本，是结构化数据。

## 回收：做完了不代表消失

V2 子 agent 完成后不会被立即销毁。它进入"resident"状态，保留在内存中，可以被后续消息重新激活。

但内存有限。当需要 spawn 新子 agent 但容量已满时，系统用 LRU 驱逐：从最久未使用的开始，检查它是否可卸载（状态已终结、没有活跃 turn、没有排队消息）。三个条件缺一不可——不会在子 agent 还在工作时把它杀掉。

如果所有候选者都不可卸载，返回 `AgentLimitReached` 错误。父 agent 会看到"无法创建新子 agent"，然后自己决定怎么办。

## 通知没有过期机制

子 agent 的完成通知被注入父 agent 后，它就变成了父 agent 对话历史的一部分。没有"已读后删除"，没有"N 步后自动过期"。它和所有其他历史消息一样，靠 compaction 统一处理。

这意味着：如果父 agent 和很多子 agent 交互频繁，完成通知会不断累积，直到触发 compaction 被摘要压缩。系统不为通知设计独立的清理逻辑——复用已有的 compaction 机制，少一套代码。

## 为什么不用 RPC？

因为 RPC 要求父 agent "等待"子 agent。但 Codex 的 Session 是串行的——一个 Session 同一时刻只能跑一个 turn。如果父 agent 阻塞等待子 agent，它就什么都做不了了。

消息传递模型让父 agent 完全解耦：它不需要知道子 agent 的内部结构，不需要等待，不需要处理回调。它只需要在下次处理输入时，看到收件箱里多了一条消息。

代价是：父 agent 不能"同步等待子 agent 的结果"。如果你需要"父 agent 阻塞直到子 agent 返回"，这个设计不适合你。但 Codex 的场景是异步的：父 agent 继续工作，子 agent 结果稍后到达。就像你给同事布置任务后不会站在旁边等，而是回去干自己的事。

子 agent 的结果不是"返回"给父 agent，而是"发生"在父 agent 的输入流里。这一句话就是整个多 agent 通信的设计哲学。
