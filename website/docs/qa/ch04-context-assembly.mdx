## 设计问题

每次调模型之前，system prompt 里应该放什么？是一个固定字符串，还是动态拼接的？如果是动态的，谁决定拼什么、按什么顺序拼？

## 直觉答案 vs 实际选择

直觉上，system prompt 是一个写死的模板，加上一些变量替换。简单、可预测。

Codex 的实际选择是：**system prompt 不存在。模型看到的"上下文"是一套装配流程的产物——每个 model step 都从多个来源动态拼接，而且不是每步都全量重建。**

## 装配流程：不是一次性的，是每步的

```rust
// codex-rs/core/src/session/mod.rs:2908
pub(crate) async fn capture_step_context(
    self: &Arc<Self>,
    turn_context: Arc<TurnContext>,
) -> Arc<StepContext> {
    let environments = turn_context.environments.refresh_readiness();
    self.services.agents_md_manager.refresh(&turn_context.config, &environments).await;
    let loaded_agents_md = self.services.agents_md_manager.get_loaded().await;
    let mcp = self.mcp_runtime_for_step(...).await;
    Arc::new(StepContext::new(
        turn_context, environments, selected_capability_roots,
        executor_capability_discovery, mcp, loaded_agents_md,
    ))
}
```

每个 model step 之前，`capture_step_context` 重新收集：

- AGENTS.md（可能因为 cd 到新目录而变化）
- MCP 工具列表（可能因为 server 启动/关闭而变化）
- 环境选择（workspace roots、cwd）
- 能力发现（executor capabilities）

这意味着模型在同一个 turn 的不同 step 里，看到的工具集和上下文可能不同。

## 首次全量 vs 后续增量

```rust
// codex-rs/core/src/session/mod.rs:3576
let (context_items, world_state_item) = if should_inject_full_context {
    // 首次：全量注入（base instructions + world state + MCP tools）
    let context_items = self.build_initial_context_with_world_state_and_mcp(...).await;
    state.history.set_world_state_baseline(snapshot);
    (context_items, Some(WorldStateItem::full(snapshot)))
} else {
    // 后续：只注入变化部分
    let context_items = self.build_settings_update_items(...).await;
    let (fragments, _) = state.history.update_world_state(world_state);
    context_items.extend(fragments);
    (context_items, world_state_item)
};
```

关键设计：

- **首次**（`reference_context_item.is_none()`）：全量注入所有上下文，设为 baseline
- **后续**：只注入和 baseline 的 diff（配置变化、world state 变化、turn context 变化）

这不是为了省 token——是为了**保护 prompt cache 前缀**。如果每步都全量重建，前缀变了，cache 就 miss。只追加 diff，前缀不变，cache 命中。

## 上下文的来源清单

从 `context/` 目录的文件名可以看出所有注入来源：

| 来源 | 文件 | 注入时机 |
|------|------|----------|
| 基础指令 | `user_instructions.rs` | 首次全量 |
| AGENTS.md | `world_state/agents_md.rs` | 每步检查变化 |
| 权限说明 | `permissions_instructions.rs` | 配置变化时 |
| 环境信息 | `environment_context.rs` | 每步（git status 等） |
| 可用 skills | `available_skills_instructions.rs` | 首次或变化时 |
| 可用插件 | `available_plugins_instructions.rs` | 首次或变化时 |
| MCP 工具 | 通过 StepContext.mcp | 每步重建 |
| 子 agent 列表 | `world_state/environment.rs` | 每步（baseline/diff） |
| 时间提醒 | `current_time_reminder.rs` | 周期性 |
| 多 agent 指令 | `multi_agent_mode_instructions.rs` | 首次 |

## 为什么不是"一个 system prompt"

传统做法是把所有信息塞进一个 system prompt 字符串。Codex 不这样做，因为：

1. **prompt cache 要求前缀稳定**——如果 system prompt 每步都变（比如包含 git status），cache 永远 miss
2. **增量注入让变化可追踪**——rollout 里记录的是"第 N 步注入了什么 diff"，而不是"第 N 步的完整 prompt 是什么"
3. **不同来源有不同的更新频率**——base instructions 整个 session 不变，git status 每步都变，AGENTS.md 偶尔变

## 取舍

**得到什么**：

- prompt cache 命中率最大化——前缀稳定，只有尾部追加
- 上下文变化可审计——rollout 里有完整的注入记录
- 灵活——新来源只需要实现一个 fragment builder，不需要改"主模板"

**放弃什么**：

- 最终 prompt 难以预测——你无法打开一个文件看到"模型看到的完整 system prompt"
- 调试困难——"为什么模型不知道 X？"需要追踪 X 的注入条件和 diff 逻辑
- 实现复杂——baseline/diff 机制比"每次全量拼"复杂得多

## 对读者的启示

如果你的 agent 系统不需要 prompt cache（比如用本地模型），全量重建 system prompt 是最简单的方案。但如果你用 OpenAI/Anthropic 的 API 并且关心成本，prompt cache 就是必须考虑的约束。

Codex 的核心洞察是：**prompt cache 的命中条件是"前缀不变"，所以变化的信息必须追加在尾部，不能插入前缀。** 这个约束反过来决定了整个上下文装配的架构——不是"拼一个字符串"，而是"维护一个 baseline + 追加 diff 序列"。
