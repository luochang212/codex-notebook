## 设计问题

每个 model step 之前，环境信息（git status、子 agent 列表、权限配置、AGENTS.md）都可能变化。如果每步都全量注入这些信息，token 浪费严重且破坏 prompt cache。如果完全不注入，模型就不知道环境变了。怎么平衡？

## 直觉答案 vs 实际选择

直觉上，每步把最新的环境信息拼进 system prompt。简单，但每步都变的前缀让 prompt cache 永远 miss。

Codex 的实际选择是：**WorldState 每步重建，但记录到模型上下文时采用 baseline/diff 机制——首次全量写入，后续只追加变化部分。**

## WorldState 的组成

```rust
// codex-rs/core/src/session/world_state.rs:19
pub(crate) async fn build_world_state_for_step(&self, step_context: &StepContext) -> WorldState {
    let mut world_state = WorldState::default();
    world_state.add_section(RealtimeState::new(...));
    world_state.add_section(AgentsMdState::new(...));
    world_state.add_section(PermissionsState::new(...));
    world_state.add_section(EnvironmentsState::from_turn_context_with_environments(...)
        .with_subagents(environment_subagents));
    world_state.add_section(AppsInstructionsState::new(...));
    world_state.add_section(PluginsInstructionsState::new(...));
    // extension 贡献的额外 section
    for contributor in self.services.extensions.context_contributors() {
        for section in contributor.contribute_world_state(...).await {
            world_state.add_extension_section(section);
        }
    }
    world_state
}
```

WorldState 是一个 section 集合，每个 section 对应一类环境信息。每步都从头构建——但构建不等于注入。

## baseline/diff：只记录变化

```rust
// codex-rs/core/src/context_manager/history.rs:91
pub(crate) fn update_world_state(
    &mut self,
    world_state: &WorldState,
) -> (Vec<Box<dyn ContextualUserFragment>>, Option<WorldStateItem>) {
    let snapshot = world_state.snapshot();
    // 和 baseline 做 diff，只渲染变化部分
    let fragments = world_state.render_history_diff(
        self.world_state_baseline.as_ref(), &self.items);
    // rollout 记录：首次全量，后续 patch
    let rollout_item = self.world_state_baseline.as_ref().map_or_else(
        || Some(WorldStateItem::full(snapshot.clone().into_value())),
        |previous| snapshot.merge_patch_from(previous).map(WorldStateItem::patch),
    );
    self.world_state_baseline = Some(snapshot);
    (fragments, rollout_item)
}
```

两层记录：

1. **模型可见层**（`fragments`）：`render_history_diff` 只生成和 baseline 不同的 section 文本，追加到对话历史尾部
2. **持久化层**（`rollout_item`）：首次记录 `WorldStateItem::full`，后续记录 `WorldStateItem::patch`（JSON merge patch）

如果某步环境完全没变（比如模型只是输出了一段文本），`fragments` 为空，不注入任何内容。

## 为什么不是每步全量

假设一个 turn 有 10 个 step（模型调了 10 次工具）。如果每步全量注入 WorldState：

- 10 × ~2000 token = 20000 token 浪费在重复的环境信息上
- 每步前缀都变（因为追加了新的全量 WorldState），prompt cache 永远 miss

用 baseline/diff：

- 首次 ~2000 token，后续 9 步可能只有 ~50 token（比如只有 git status 变了）
- 前缀不变，prompt cache 持续命中

## 取舍

**得到什么**：

- token 效率——不变的信息不重复注入
- prompt cache 友好——前缀稳定
- 持久化紧凑——rollout 里只存 patch，不存每步的全量快照

**放弃什么**：

- 实现复杂度——需要维护 baseline、做 diff、处理 baseline 重置（compaction 后）
- 调试困难——"模型看到的完整环境信息"需要从 baseline + 所有 patch 重建
- baseline 丢失风险——如果 compaction 时 baseline 处理不当，后续 diff 可能基于错误的基准

## 对读者的启示

如果你的 agent 每个 turn 只有 1-2 个 step，baseline/diff 的收益很小，全量注入更简单。但如果你的 agent 一个 turn 可能跑 20+ 个 tool call（coding agent 的典型场景），环境信息的重复注入会成为 token 消耗的大头。

核心洞察：**"每步重建"和"每步注入"是两件事。** 重建是为了保证正确性（不遗漏变化），注入是为了效率（只告诉模型新东西）。把这两件事分开，才能同时满足正确性和效率。
