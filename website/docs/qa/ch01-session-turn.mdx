## 设计问题

一个 coding agent 的"一次对话"应该由谁管理？是调用方（CLI、IDE 插件）管，还是 agent 内核管？

## 直觉答案 vs 实际选择

直觉上，调用方发送一条消息、等待一条回复，对话就结束了。生命周期由调用方控制，agent 只是一个无状态的请求-响应服务。

Codex 的实际选择是：**Session 是一个长生命周期的有状态对象，拥有自己的事件循环；Turn 是 Session 内部的一个有界执行单元。** 调用方不控制 turn 的开始和结束——它只是往 Session 的 submission channel 里投递一个 `Op::UserInput`，然后从 event channel 里读取事件流。

```rust
// codex-rs/core/src/session/session.rs:28
pub(crate) struct Session {
    pub(crate) thread_id: ThreadId,
    pub(super) tx_event: Sender<Event>,
    pub(super) agent_status: watch::Sender<AgentStatus>,
    pub(super) state: Mutex<SessionState>,
    pub(crate) conversation: Arc<RealtimeConversationManager>,
    pub(crate) active_turn: Mutex<Option<ActiveTurn>>,
    pub(crate) input_queue: InputQueue,
    pub(crate) guardian_review_session: GuardianReviewSessionManager,
    pub(crate) services: SessionServices,
    // ...
}
```

Session 创建时启动一个后台 tokio task（`submission_loop`），这个 task 就是 Session 的"心跳"：

```rust
// codex-rs/core/src/session/handlers.rs:695
pub(super) async fn submission_loop(
    sess: Arc<Session>,
    config: Arc<Config>,
    rx_sub: Receiver<Submission>,
) {
    while let Ok(sub) = rx_sub.recv().await {
        match sub.op {
            Op::UserInput { .. } => {
                user_input_or_turn(&sess, sub.id, sub.op, ...).await;
            }
            Op::Interrupt => { interrupt(&sess).await; }
            Op::Shutdown => { /* 退出循环 */ }
            // ...
        }
    }
}
```

调用方和 Session 之间只有两个 channel：`tx_sub`（投递操作）和 `rx_event`（接收事件）。调用方不能"等待回复"——它只能订阅事件流，自己判断 turn 是否结束。

## Turn：Session 内部的有界循环

一个 Turn 从 `run_turn` 开始，到模型不再请求工具调用（或发生错误/中断）时结束：

```rust
// codex-rs/core/src/session/turn.rs:144
pub(crate) async fn run_turn(
    sess: Arc<Session>,
    turn_context: Arc<TurnContext>,
    input: Vec<TurnInput>,
    cancellation_token: CancellationToken,
) -> CodexResult<Option<String>> {
    // 1. 预压缩（如果上下文快满了）
    run_pre_sampling_compact(&sess, &turn_context, ...).await?;
    // 2. 捕获 step context，记录 world state
    let first_step_context = sess.capture_step_context(...).await;
    // 3. 主循环
    loop {
        let step_context = sess.capture_step_context(...).await;
        let result = run_sampling_request(...).await;
        match result {
            Ok(output) if output.needs_follow_up => continue,  // 模型还要调工具
            Ok(_) => break,                                     // 模型说完了
            Err(TurnAborted) => return Err(...),                // 被中断
            Err(e) => { /* 发错误事件 */ break; }
        }
    }
}
```

关键设计：`needs_follow_up` 为 true 意味着模型请求了工具调用，工具已执行，结果已回注历史，循环继续。为 false 意味着模型只输出了文本，turn 结束。

## 为什么 Session 和 Turn 必须分开

一个 Session 可以有多个 Turn。Turn 之间共享：

- 对话历史（`conversation`）
- 配置（`SessionConfiguration`）
- 审批状态（`guardian_review_session`）
- 输入队列（`input_queue`）

但每个 Turn 有自己独立的：

- `TurnContext`（模型信息、sub_id、terminal_error）
- `CancellationToken`（可以被独立中断）
- `StepContext`（每步重建的环境快照）

这意味着：用户可以在一个 turn 结束后发新消息开启新 turn，历史自然延续；也可以在 turn 进行中通过 `Op::Interrupt` 中断当前 turn，Session 本身不死。

## 取舍

**得到什么**：

- 跨 turn 的上下文连续性——不需要调用方自己管理历史
- 中断恢复——中断一个 turn 不杀 Session，用户可以继续对话
- 持久化——Session 的 rollout 可以序列化到磁盘，重启后恢复
- 多 agent——子 agent 就是一个独立的 Session，父 agent 通过 channel 和它通信

**放弃什么**：

- Session 是重量级对象（Mutex、watch channel、Arc 引用计数），不适合"一问一答"的轻量场景
- 调用方失去了对 turn 边界的直接控制——不能"等这条消息的回复"，只能订阅事件流自己判断
- submission_loop 是单线程串行的——一个 Session 同一时刻只能跑一个 turn

## 对读者的启示

如果你要做一个 coding agent，第一个设计决策不是"用什么模型"，而是"谁拥有对话的生命周期"。

- 如果调用方拥有（无状态 API），你得到简单性，但失去跨 turn 连续性、中断恢复、持久化。
- 如果内核拥有（有状态 Session），你得到上述所有能力，但必须设计好 channel 协议和事件流格式，让调用方能"观察"而非"控制"。

Codex 选了后者。这个选择让 TUI、IDE 插件、子 agent 都可以用同一套 Session 接口，只是订阅事件流的方式不同。
