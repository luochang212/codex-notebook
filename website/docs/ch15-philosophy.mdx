写完前 14 章，我回头翻了一遍，想看看这些独立的设计决策背后有没有一条统一的线。

有。

## 一个问题

整本书只回答一个问题：**如何在沙箱安全约束下，让 LLM 拥有尽可能大的代码修改自主权？**

注意这个问题的结构。它不是"如何让 LLM 写代码"——那太简单了，给一个 shell 工具就行。它也不是"如何保证安全"——那也太简单了，什么都不让做就行。

它是一个约束优化问题。在"安全"这个硬约束下，最大化"自主权"。这两个方向是矛盾的。安全要求收紧边界，自主要求放开边界。所有 14 章的设计决策，都是在这条张力线上找平衡点。

## 回看每一章的平衡点

ch01 的 Session/Turn：自主权意味着"模型可以连续执行多步操作"，安全意味着"用户随时可以中断，中断不丢状态"。平衡点是：Turn 有界，Session 无限。

ch02 的模型调用：自主权意味着"网络断了不能丢进度"，安全意味着"不能无限重试浪费资源"。平衡点是：三层降级，有上限。

ch03 的工具循环：自主权意味着“模型可以自己决定下一步做什么”，安全意味着“不能无限循环”。平衡点是：RwLock 门控并行，compaction 保证窗口不爆，没有硬步数上限。

ch04-06 的上下文工程：自主权意味着"模型看到的信息越多越好"，安全（这里是成本约束）意味着"context window 有限"。平衡点是：baseline + diff，只告诉模型变化了什么。

ch07-08 的工具系统和沙箱：自主权意味着"模型可以执行任意命令"，安全意味着"不能碰坏文件系统"。平衡点是：OS 级沙箱隔离进程，exec policy 过滤命令。

ch09 的 Guardian：自主权意味着"不要每条命令都问我"，安全意味着"危险操作必须有人看"。平衡点是：一个专门的模型做意图级判断，只在不确定时升级给人类。

ch10-11 的多 agent：自主权意味着"复杂任务可以分治并行"，安全意味着"不能无限 spawn、不能失控"。平衡点是：深度限制 + LRU 驱逐 + 父 agent 是唯一的协调者。

ch12-13 的 skill 和 MCP：自主权意味着"能力可以扩展"，安全意味着"扩展不能破坏内核"。平衡点是：skill 是文档（零运行时风险），MCP 是独立进程（崩溃不影响主进程）。

ch14 的配置层：自主权意味着"每个利益相关方都能定制行为"，安全意味着"企业管控不能被用户绕过"。平衡点是：数值优先级，越具体的覆盖越通用的。

## 内核只做三件事

抽掉所有扩展和约束，Codex 的内核（`codex-rs/core`）只做三件事：

1. **管生命周期**：Session 拥有状态，Turn 是有界执行单元
2. **构造上下文**：每步装配 system prompt + environment + world state
3. **执行工具**：ToolRouter 路由，结果回注 turn 循环

其他一切都是这三件事的扩展或约束。沙箱约束工具执行。Guardian 约束审批。MCP 扩展工具集。Skill 扩展上下文。多 agent 扩展执行容量。配置层定制行为。

内核小，扩展多。这是"受控自主"在代码结构上的体现：内核提供自主执行的能力，扩展层提供控制和定制的手段。

## 安全不是事后加的

很多 agent 系统的安全模型是"先做功能，再加审批"。Codex 不是。

沙箱是执行的前提条件。sandbox policy 和 approval policy 是两个独立维度，它们的组合决定了 agent 的自主程度。宽松的 approval policy 不是“去掉所有安全检查”，而是“把安全检查从交互层移到基础设施层”。用户不再逐条审批，但进程仍然被隔离，网络仍然被过滤，命令仍然被 exec policy 检查。

安全等级不是"功能多少"的问题，是"边界在哪"的问题。

## 这套架构不适合什么

说完适合的，说不适合的。这同样重要。

纯文本生成不需要这套架构。不需要沙箱、不需要工具执行、不需要多 agent。用 Codex 写一首诗，就像开坦克去买菜。

实时交互也有支持。源码中有完整的 realtime conversation handlers（audio、speech、text、start、close）。Turn 模型是主路径，但不是唯一路径。而且即使在 turn 内，用户也可以通过 steer 机制注入新输入，不必等 turn 结束。

极简场景是过度设计。如果你只需要"LLM + 一个 bash 工具"，多层配置、多 agent 协议、LRU 驱逐都是负担。

还有一个根本限制：沙箱有历史漏洞，Guardian 是 LLM 会误判。这套架构提供的是"合理的安全边界"，不是"绝对安全"。如果你的威胁模型是"对抗恶意模型输出"，还需要额外的隔离层。

## 如果要二次开发

五个扩展点，按风险从低到高：

1. **加工具**：实现 `CoreToolRuntime` trait，注册到 ToolRouter。最常见，风险最低。
2. **改上下文**：修改 `build_initial_context` 或添加新的 WorldState section。影响模型"看到什么"。
3. **改审批**：修改 exec policy 规则或 guardian prompt。影响"什么操作需要确认"。
4. **加角色**：在 `built_in::configs()` 里加一个 `AgentRoleConfig`。影响多 agent 的分工。
5. **改 compaction**：修改触发条件或摘要策略。影响长对话的质量。

不建议改的：Session/Turn 生命周期、配置层合并逻辑、沙箱实现。这些是承重墙。动了它们，上面所有层的行为都会变。

## 最后一个观察

Codex 用 Rust 写核心，也用 Rust 写 CLI（`codex-cli` crate）。IDE 集成和 TUI 是另外的前端。

这不是技术偏好。agent 运行时是长生命周期进程。一个 turn 可能跑 10 分钟，一个 session 可能跑 2 小时。内存泄漏、data race、panic 导致的崩溃——这些在短生命周期的 Web 服务里可以容忍（重启就行），在 agent 运行时里意味着用户的工作丢了。

Rust 的类型系统和所有权模型在编译期消灭了这类问题。代价是写得慢。但对于"基础设施"级别的软件，运行时可靠性比开发速度重要。你写一次，它跑一百万次。
