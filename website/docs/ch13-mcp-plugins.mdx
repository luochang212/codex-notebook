上一章的结论是：skill 只能注入知识，不能执行代码。你写一份 `SKILL.md` 告诉模型"部署时先跑 migration 再切流量"，模型确实会按这个顺序来。但它跑不了 migration。它没有手。

知识用文档就够了。能力需要进程。

这一章讲的就是"手"的问题：怎么让 agent 调用 GitHub API、操作数据库、执行自定义部署脚本——同时不让这些外部代码碰到 agent 的内部状态？

## 一个进程边界

答案很朴素：把第三方工具放在独立的运行时里。

MCP server 有两种 transport。Stdio transport 启动一个本地子进程，通过标准输入输出通信。StreamableHttp transport 连接一个远程 HTTP 服务，不是 Codex 创建的进程。两种情况下，agent 说的都是 JSON-RPC：「调用这个工具，参数是这些」。server 干完活，把结果用 JSON 回传。

agent 不知道 server 的实现语言。server 不知道 agent 的内部结构。它们之间只有一份合同：tool schema（name + description + parameters）。

这跟微服务的思路一样。但比微服务简单——没有服务发现、没有负载均衡、没有熔断。一个 agent 对一组 server，点对点。

## 300 个工具的问题

假设你接了 30 个 MCP server，每个提供 10 个工具。300 个 tool spec 塞进模型的 tool list，光 schema 就占几万 token。模型还没开始干活，context window 已经被工具说明书吃掉了。

Codex 的解法分两档：

**Direct 模式**：工具的完整 schema 直接出现在 tool list 里。模型每次调用都能看到所有工具。适合工具少的场景。

**Deferred 模式**：工具不直接出现。模型先看到一个搜索入口，按需查找，找到后再加载具体 schema。300 个工具变成了"一个搜索框 + 按需加载"。

判断用哪种模式的逻辑在 `build_mcp_tool_runtimes` 里，一行分支：如果启用了 search tool，走 Deferred；否则走 Direct。

## 不是所有工具都对模型可见

MCP server 注册的工具里，有些是给编排层用的（比如健康检查、版本查询），模型不需要看到。每个工具的 meta 里可以标记 model 可见性。不可见的工具仍然存在于连接中，但不会进入模型的 tool list。

这层过滤发生在 agent 侧，不在 server 侧。server 可以注册 20 个工具，agent 只暴露其中 5 个给模型。

## 审批不因为是外部的就更宽松

一个常见误解：外部工具应该被更严格地审批。

Codex 没有这样做。但 MCP 工具也不是走和内建 shell 相同的 exec policy 链路。MCP 有自己独立的审批体系：tool approval mode、app policy、annotation 判断、session/persistent remembered approval、permission hook，以及可选的 Guardian/user elicitation。不因为是“外部的”就加一层，也不因为“已经通过 MCP 协议了”就减一层。

这背后有一个判断：审批的对象是**操作的后果**，不是**工具的来源**。`git push --force` 不管是内建工具调的还是 MCP 工具调的，后果一样。但审批的实现路径不同——MCP 有自己的 approval mode 配置，不是先拿命令去匹配 exec policy。

但有一个 MCP 特有的安全边界值得注意：审批能控制"是否调用"，不能控制"调用结果是否可信"。一个恶意 MCP server 可以返回误导性结果——比如告诉你"测试全部通过"但其实没跑。agent 没有能力验证外部 server 的返回值。这是信任边界。

## required server：启动失败就是 fatal

`McpConnectionManager` 管理所有活跃的 server 连接。每个 server 有一个标记：required 或 optional。

required server 启动失败，整个 session 报错，不进入对话。optional server 启动失败，静默跳过，后续调用时再报错。

为什么区分？因为有些工具是任务的前提。如果你的部署流程依赖一个数据库 MCP server，它没起来就开始对话，模型会尝试调用一个不存在的工具，得到错误，重试，再错误——浪费 token 和用户时间。不如一开始就说清楚：这个 server 没起来，今天干不了活。

## 插件：MCP 的打包分发

`PluginsManager` 在 MCP 之上加了一层薄包装：一个 plugin 可以包含多个 MCP server、多个 skill、apps、hooks 和 interface 信息。

它不是另一种协议。它解决的是“怎么把 MCP server + skill + hooks 打包给用户安装”的问题。用户不需要手动编辑 config 文件写 server 地址和参数——安装一个 plugin 就行。

skill + MCP server 的组合也自然：skill 告诉模型"什么时候该用这个工具、用的时候注意什么"，MCP server 提供实际执行能力。知识 + 能力，文档 + 进程。

## 代价

Stdio transport 下，每次工具调用都跨进程。延迟在毫秒级。StreamableHttp transport 下，延迟在几十毫秒级。对于“读一个文件”这种操作，内建工具微秒级完成，MCP 工具要多走一圈。

调试变难了。问题可能在 agent 侧（路由错误）、MCP client 侧（序列化失败）、MCP server 侧（逻辑 bug）。三层栈，每层有自己的日志。

Stdio transport 下，每个 server 是独立进程，有启动时间。10 个 server 的冷启动可能需要几秒。StreamableHttp transport 没有这个问题——远端服务已经在跑了。

换来的：任何语言都能写 MCP server。server 崩溃不影响 agent 主进程。工具生态可以独立于 agent 版本演进。

一句话：MCP 用进程隔离换来了语言无关和故障隔离，代价是每次调用多一跳。对于"执行能力"这个需求，这一跳值得。
