模型在一次响应里输出了 4 个 tool call：3 个 `read`（读三个不同的文件）和 1 个 `shell`（跑 `cargo test`）。

系统现在面对一个很具体的问题：这 4 个操作，先跑哪个？同时跑还是排队？如果 `cargo test` 跑到一半用户按了 Ctrl+C，那 3 个 read 的结果怎么办？如果 `cargo test` 返回了非零退出码，整个 turn 是崩了还是继续？

这些问题的答案，藏在一个 `RwLock<()>` 里。

## 读可以并发，写必须独占

3 个 `read` 互不干扰——它们只是读文件，不改任何东西。让它们排队等是浪费。但 `shell` 不一样：`cargo test` 会写 target/ 目录、可能改文件锁、可能起子进程。如果它和另一个写操作同时跑，结果不可预测。

Codex 用了一个极其简洁的门控：一把 `RwLock<()>`。锁里面没有数据，就是一个空元组——它不保护任何资源，只保护**执行权**。

支持并行的工具（read、grep、web_search）拿 read lock。多个 read lock 可以同时持有。不支持并行的工具（shell、apply_patch）拿 write lock。Write lock 独占，必须等所有 read lock 释放。

所以那 4 个 tool call 的执行顺序是：3 个 read 同时跑，shell 等它们全部完成后再开始。不需要调度器，不需要优先级队列。RwLock 的语义天然表达了"读可以并发，写必须独占"。

代价是：即使模型想并行 apply 两个不冲突的 patch（改两个完全不同的文件），也必须排队。系统不做冲突检测，一律串行。简单，但保守。

## 工具失败不是灾难

`cargo test` 跑完了，退出码 1。测试失败了。

在大多数人的直觉里，"命令失败"应该触发某种错误处理——重试？回滚？报告用户？

Codex 的选择更有趣：**把错误信息当成工具的正常输出，塞回对话历史，让模型自己决定下一步。**

测试失败的 stderr 变成了 tool result 的一部分。模型看到"test result: FAILED. 3 passed; 2 failed"，然后自己判断：是改代码再跑一次，还是换个思路。Turn 循环不中断。

只有一种情况会真正杀掉 turn：`Fatal` 错误——比如沙箱进程本身崩溃了。普通的"命令返回非零""文件不存在""权限不够"，都是信息，不是灾难。

这个设计的含义是：**模型被允许失败。** 它不需要第一次就做对。它可以试、可以错、可以调整。这比"失败就停"的 agent 灵活得多。

## Ctrl+C 的两种处理方式

用户按了 Ctrl+C。此时 `cargo test` 正在跑，已经输出了 30 行日志。

系统面临一个选择：是立刻杀掉这个进程（丢失部分输出），还是给它一个机会优雅退出（收集已有输出）？

Codex 两种都做，取决于工具类型：

- **shell 命令**：发 SIGTERM，等进程退出，收集部分输出。因为 shell 命令可能有子进程、可能写了临时文件，直接 abort 可能留下垃圾。
- **纯计算工具**（比如内部的文本处理）：直接 abort tokio task。没有副作用，杀了就杀了。

这个区分由工具自己声明（`wait_for_runtime_cancellation`），不是调度器猜的。

## turn 循环只看到一个布尔值

说了这么多细节，但 turn 循环本身——ch01 里那个 `loop { ... }`——对这一切完全无感。它只看到一个布尔值：`needs_follow_up`。

`true` 意味着：模型输出了 tool call，工具已执行，结果已回注历史，需要再调一次模型。`false` 意味着：模型只输出了文本，turn 结束。

并行门控、错误处理、中断策略——全部封装在 `run_sampling_request` 内部。Turn 循环不知道也不关心工具是怎么跑的。它只知道"模型还想继续"或者"模型说完了"。

这种封装的好处是：你可以换掉整个工具执行层（比如换成远程执行、换成容器化执行），turn 循环一行不用改。坏处是：调试"为什么这个工具跑了 30 秒"时，你得穿透好几层抽象才能看到真相。
