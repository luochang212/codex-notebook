title: 为什么不给 LLM 一个万能 bash 就够了？

---

这个问题我被人问过很多次："LLM 已经会写 shell 命令了，为什么还要设计一套结构化工具？直接给它一个 bash，它想做什么就写什么命令，不就什么都能做了？"

听起来很有道理。一个万能工具，零设计成本，无限灵活性。为什么 Codex 不这样做？

因为它试过。然后发现三个问题。

## 问题一：模型会写错文件

让模型用 `echo "..." > file.rs` 写文件，它必须输出整个文件的内容。一个 500 行的文件改 2 行，它得输出 500 行。而模型会幻觉——它可能"记得"文件里有一个实际上不存在的函数，然后自信地把它写进去。

更糟的是：全量覆写是静默的。即使模型记错了文件内容，`echo >` 也会成功执行。错误不会被发现，直到用户跑测试。

Codex 的解法是 `apply_patch`：不是"写入文件"，而是"在文件中找到这段文本，替换成那段文本"。如果模型幻觉了一个不存在的函数，精确替换会失败——找不到 old text，操作报错。错误在发生的那一刻就被捕获了，而不是等到用户跑测试时才发现。

而且模型只需要输出变化的 2 行，不是整个 500 行。省 token，也省幻觉的机会。

## 问题二：审批无法精确到操作级别

如果只有一个 bash 工具，审批系统只能看到"模型要执行一条命令"。但一条命令可能同时做安全的事（`cargo build`）和不安全的事（`curl | bash`）。你没法对一条复合命令做细粒度审批。

结构化工具让审批可以精确到操作类型：`read` 永远安全，直接放行；`apply_patch` 需要检查目标文件；`shell` 需要过 exec policy。不同工具走不同的审批路径。

如果一切都是 bash，你就只能对每条命令做统一的字符串匹配——脆弱、容易被绕过、而且用户体验很差（每条命令都弹框）。

## 问题三：工具 spec 是行为引导

这是最微妙的一点。工具的 description 不是写给人看的文档——它是写给模型看的行为指令。

`apply_patch` 的 spec 很简短：它告诉模型这个工具用于编辑文件，格式是 FREEFORM，不要包在 JSON 里。行为引导（“先 read 再 patch”“不要猜测文件内容”）不在工具 spec 里，而在 system prompt 的指令中。工具 spec 定义“这个工具是什么、参数怎么传”，system prompt 定义“应该怎么用”。

一个 bash 工具的 spec 能写什么？"请安全地使用 shell"？这等于没说。结构化工具的 spec 是具体的、可操作的行为引导，而不是笼统的"注意安全"。

## bash 仍然存在

说了这么多，bash 并没有被取消。它仍然是 Codex 的工具之一。但它的定位变了：不是默认路径，而是"其他工具做不了的事"的出口。

`cargo build`、`git status`、`npm test`——这些事结构化工具做不了，必须走 shell。但文件读写、搜索、patch——这些高频操作走结构化工具，因为更精确、更安全、更省 token。

分层是：结构化工具为主，bash 兆底，沙箱强制安全。工具 spec 引导模型"应该怎么做"，沙箱和 exec policy 强制模型"不能怎么做"。两层防线，缺一不可。

## 代价

10 个工具比 1 个 bash 难选。模型的选择负担增加了。每个工具的 schema + description 都消耗 context window。有些事用 bash 一行搞定，用结构化工具要好几步。

但这些代价是可控的：工具 spec 占的 token 是固定的（不随对话长度增长），而"模型写错文件"的代价是不可控的（用户可能几小时后才发现）。

安全不应该依赖模型的自觉，而应该由系统强制执行。这是结构化工具存在的根本理由。
