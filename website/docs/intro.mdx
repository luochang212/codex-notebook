## Codex 不是一个 LLM 包装器

如果你打开 `openai/codex` 的源码，期待看到"调用 OpenAI API + 拼接 prompt + 显示结果"，你会失望。Codex 的核心不是一个 chatbot，而是一个**受控执行运行时**（controlled execution runtime）。

它管理 agent 的完整生命周期：session 创建、turn 执行、工具调度、上下文压缩、子 agent 回收、沙箱隔离、审批流程。LLM 调用只是其中一个环节——和"启动一个沙箱进程"或"压缩 200 轮对话历史"同等重要。

## 和同类系统的本质区别

Cursor 是一个 IDE，Aider 是一个 CLI 工具，Claude Code 是一个终端 agent。Codex 和它们的区别不在于"功能多少"，而在于**安全模型**：

- Codex 的执行发生在 OS 级沙箱内（bubblewrap / seatbelt / landlock）
- 沙箱不是"可选的安全加固"，而是"全自动执行"的前提条件
- 没有沙箱，Codex 就退化成一个需要逐条审批的命令建议器

这意味着 Codex 的架构复杂度有很大一部分花在"如何让模型自由行动而不出事"上。

## 全书主线

这本书沿着一条设计张力线展开：

> **如何在沙箱安全约束下，让 LLM 拥有尽可能大的代码修改自主权？**

Codex 的回答是一套"受控自主"架构：

- Session 管理生命周期 → 第一篇
- 上下文工程决定模型看到什么 → 第二篇
- 工具系统决定模型能做什么 → 第三篇
- 沙箱和 Guardian 决定边界在哪 → 第三、四篇
- 多 agent 把自主权分治 → 第五篇
- skill、MCP、配置层让用户定制行为 → 第六篇
- 设计哲学总结 → 第七篇

## 怎么读这本书

每一章回答一个设计问题。结构统一：

1. **设计问题**：不是"这个模块有什么"，而是"为什么要这样设计"
2. **直觉答案 vs 实际选择**：先说你可能会怎么想，再说 Codex 实际怎么做的
3. **源码佐证**：代码只在需要时出场，只展示设计核心
4. **取舍**：每个决策得到什么、放弃什么
5. **对读者的启示**：如果你要做类似系统，哪些可以借鉴

你不需要从头读到尾。每章是独立的，但后面的章节会引用前面的概念。建议至少先读第一篇（循环引擎），它建立了全书的技术基础。

有工程经验、想理解 coding agent 系统设计的开发者。你不需要会 Rust（代码会解释），但需要理解进程、文件系统、网络这些 OS 概念。
