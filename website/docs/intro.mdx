我第一次打开 `openai/codex` 的源码时，以为会看到一个 chatbot：调 API、拼 prompt、显示结果。大概两三百行就能搞定。

然后我看到了 397 行的 `role.rs`、236 行的 `residency.rs`、一整个 `guardian/` 目录、一个用 Rust 写的 OS 级沙箱适配层。

这不是一个 chatbot。这是一个**运行时**。

## 一个运行时和一个工具的区别

Cursor 是 IDE。Aider 是 CLI。Claude Code 是终端 agent。它们都是"LLM + 一些工具"的组合——模型说"我要读这个文件"，工具读了，返回结果，模型继续。

Codex 多出来的那一层，是**生命周期管理**。

Session 什么时候创建、什么时候销毁。Turn 怎么开始、怎么中断、中断后状态怎么办。Context window 满了怎么压缩。子 agent 跑完了怎么通知父 agent。30 个 MCP server 怎么并行启动。模型连续 5 次调用失败后怎么降级。公司 IT 推送了一条"禁止访问外网"的策略，怎么保证用户覆盖不了。

这些问题，"LLM + 工具"的组合回答不了。它们需要一个系统。

## 这本书只回答一个问题

> **如何在沙箱安全约束下，让 LLM 拥有尽可能大的代码修改自主权？**

注意"约束优化"的结构。不是"如何让 LLM 写代码"——给一个 shell 工具就行。也不是"如何保证安全"——什么都不让做就行。是让这两个矛盾的方向同时成立。

15 章，每一章都是这条张力线上的一个平衡点。

## 你会读到什么

**第一篇（ch01-03）**：循环引擎。Session 拥有状态，Turn 是有界执行单元，模型在 turn 内自主决定下一步做什么。这是全书的地基。

**第二篇（ch04-06）**：上下文工程。模型每次被调用时"看到"的东西不是固定的——它是一条装配流水线的产物。前缀不能变（变了就浪费钱），增量追加，环境信息只记 diff。

**第三篇（ch07-09）**：执行与边界。模型能做什么（工具系统）、不能做什么（沙箱）、不确定的时候问谁（Guardian）。

**第四篇（ch10-11）**：多 agent。复杂任务分治并行，子 agent 完成后结果"发生"在父 agent 的收件箱里。没有 RPC，没有回调，只有信箱。

**第五篇（ch12-13）**：扩展。skill 是文档（教模型做事的方法），MCP 是进程（给模型新的能力）。知识不需要运行时，能力需要。

**第六篇（ch14）**：配置。你的配置为什么没生效？因为七层配置栈里有人优先级比你高。

**第七篇（ch15）**：回顾。14 个平衡点背后只有一条线。

## 不需要从头读到尾

每章是独立的。但后面的章节会引用前面的概念。如果只读一章，建议 ch01——它建立了"Session 拥有状态、Turn 有界"这个全书最核心的心智模型。

## 关于代码

代码只在需要证明承重逻辑时出现。你会看到 Rust，但不需要会 Rust——每段代码前会告诉你"这段要证明什么"，代码后会解释"这对结论意味着什么"。

所有源码判断基于 `openai/codex` @ `841e47b8fb`。每章底部有源码快照。

## 面向谁

有工程经验、想理解 coding agent 系统设计的开发者。你需要理解进程、文件系统、网络这些 OS 概念。不需要会 Rust，不需要用过 Codex。
