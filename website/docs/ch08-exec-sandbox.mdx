模型输出了一个 tool call：`shell("cargo build")`。

这条命令看起来很无害。但在它真正跑起来之前，它要通过多层检查。每一层解决不同粒度的问题，每一层都能独立拒绝它。

## 第一道：这条命令本身允许跑吗？

exec policy 是一个多维决策引擎。它不是简单的白名单/黑名单，而是综合多个输入做判断：

- 显式 policy rule（用户或管理员配置的 allow/deny 规则）
- 危险命令启发式（识别 `rm -rf`、`curl | bash` 这类模式）
- known-safe 启发式（识别 `git status`、`ls`、`cat` 这类只读命令）
- 当前 approval policy（用户选择的审批策略）
- permission profile 和 sandbox 形态
- 命令解析结果（是否用了复杂解析路径）

最终输出三种结果之一：`Forbidden`（直接拒绝）、`NeedsApproval`（需要审批）、`Skip`（放行）。

`cargo build` 不在 known-safe 列表里——源码测试明确断言 `cargo check` 不是 known-safe。它能否放行取决于当前 approval policy 和显式 policy rule 的组合。如果 approval policy 宽松且没有 deny 规则命中，它会被放行。如果策略严格，它可能触发审批。

这里有一个微妙的设计：审批通过后，部分情况下系统可以派生出一条 amendment（类似“以后这类命令不用再问”）。但不是所有审批都会生长规则——如果命令用了复杂解析路径（比如 heredoc fallback），自动 amendment 会被禁用，因为无法安全地泛化这条命令。

## 第二道：跑的时候能碰什么？

即使命令本身是"安全的"，它执行时也可能做危险的事。`cargo build` 会触发 build script，build script 里可以写任意代码。所以光看命令字符串不够，还得限制进程能碰什么资源。

这就是沙箱。它在 OS 级别包裹进程：

| 平台 | 技术 | 隔离内容 |
|------|------|----------|
| Linux | bubblewrap + seccomp | 文件系统、网络、系统调用 |
| macOS | seatbelt (sandbox-exec) | 文件系统、网络 |
| Windows | windows-sandbox | 文件系统、注册表 |

Linux 的默认文件系统沙箱是 bubblewrap，并在内部应用 seccomp 过滤系统调用。Landlock 是显式启用的 legacy fallback，不是默认路径。

沙箱不是不可配置的固定约束。`SandboxPolicy` 有多个变体（ReadOnly、WorkspaceWrite、DangerFullAccess、ExternalSandbox），每个变体有自己的字段。比如 WorkspaceWrite 可以配置额外可写目录，也有 `network_access` 字段控制网络是否开放。

典型配置下的核心约束：

1. 工作目录可写，其他目录只读或不可见
2. 网络由 `network_access: bool` 控制——开启就全放，关闭就全禁。没有内建的域名白名单
3. 不能 fork 出沙箱外的进程

`cargo build` 需要写 target/ 目录（在工作目录内，允许）。如果 `network_access` 为 true，它可以下载依赖；如果为 false，连接被拒绝，cargo 报错“无法下载依赖”。

但如果 build script 里写了 `std::fs::write("/etc/passwd", ...)`——沙箱拒绝。工作目录外的写操作，直接失败。进程不会崩溃，只是那个 write 调用返回 permission denied。

## 网络控制：不一定是独立的第三层

网络策略的实现方式取决于配置。`SandboxPolicy` 的每个变体都有一个 `network_access: bool` 字段——true 就开放网络，false 就禁止。没有域名级别的白名单，也没有内建的 crates.io 规则。

`cargo build` 要访问 crates.io 下载依赖。如果 `network_access` 为 true，放行。如果为 false，连接被拒绝。

多层检查的关系不是固定流水线，而是根据当前配置组合决策。exec policy 先过，然后 sandbox 包裹进程，网络控制是 sandbox policy 的一个字段。任何一层拒绝，操作就失败。

## 为什么多层而不是一层？

因为每一层解决的粒度不同。

exec policy 看的是命令字符串——它能拦截 `rm -rf /`，但拦不住 `cargo build` 里的恶意 build script。沙箱看的是进程行为——它能拦住写 /etc/passwd，但拦不住“在工作目录内写了一个错误的文件”。网络控制看的是出站连接——它能拦住访问恶意服务器，但拦不住本地文件操作。

任何单独一层都有盲区。多层叠加，盲区被覆盖。这就是纵深防御：即使一层被绕过，其他层仍然有效。

## 沙箱不是限制模型，是让模型可以被信任

这是最反直觉的一点。沙箱的存在不是为了"让模型做更少的事"，而是为了"让模型可以做更多的事"。

没有沙箱，用户很难信任 agent 的自主执行。每条命令都要人工审批，Agent 退化成了“命令建议器”。

有了沙箱，在沙箱内模型可以自由执行命令。因为即使它跑了什么奇怪的东西，沙箱会拦住真正危险的操作。

但沙箱和审批不是因果关系。sandbox policy 和 approval policy 是两个独立的维度。sandbox 决定“进程能碰什么”，approval 决定“哪些操作需要人确认”。它们的组合决定了 agent 的实际自主程度。

## 代价

三套沙箱实现需要维护——Linux、macOS、Windows 各一套。每套有不同的 API、不同的边界情况、不同的历史漏洞。bubblewrap 和 seatbelt 都不是绝对安全的——它们有过逃逸漏洞，未来可能还会有。

而且沙箱限制了模型能做的事：不能 `npm install`（如果网络被禁）、不能写工作目录外的文件、不能访问任意 URL。对于某些工作流（比如需要安装全局工具、需要访问内网服务），沙箱是碍事的。

但这个代价换来的是：用户可以信任 agent 的自主执行。没有信任，就没有自主。没有自主，coding agent 就只是一个更贵的 autocomplete。
