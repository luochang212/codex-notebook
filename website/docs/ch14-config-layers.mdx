你在 `~/.codex/config.toml` 里写了 `model = "o3"`。然后你打开一个项目，发现模型还是 o4-mini。

你检查了三遍配置文件。没有拼写错误。重启了 Codex。还是 o4-mini。

最后你在项目的 `.codex/config.toml` 里发现了一行：`model = "o4-mini"`。项目配置覆盖了你的全局配置。

这不是 bug。这是设计。但如果你不知道这套设计背后的层级结构，你会觉得自己在跟一个闹脾气的配置文件较劲。

## 不是一个文件，是一个栈

Codex 的配置不是"读一个文件"。它把所有配置来源排成一个有序栈，从低到高逐层合并。高优先级层的同名字段覆盖低优先级层的。

最终模型拿到的不是一个文件的内容，而是所有层合并后的**有效配置**。你的 `model = "o3"` 确实被读了，只是被更高优先级的层盖掉了。

## 七层，从零到五十

```rust
// codex-rs/config/src/config_layer_source.rs
pub fn precedence(&self) -> i16 {
    match self {
        ConfigLayerSource::Mdm { .. } => 0,
        ConfigLayerSource::System { .. } => 10,
        ConfigLayerSource::EnterpriseManaged { .. } => 15,
        ConfigLayerSource::User { profile, .. } => {
            if profile.is_some() { 21 } else { 20 }
        }
        ConfigLayerSource::Project { .. } => 25,
        ConfigLayerSource::SessionFlags => 30,
        ConfigLayerSource::LegacyManagedConfigTomlFromFile { .. } => 40,
        ConfigLayerSource::LegacyManagedConfigTomlFromMdm => 50,
    }
}
```

翻译成人话：

| 数值 | 层 | 谁写的 | 例子 |
|------|-----|--------|------|
| 0 | MDM | 公司 IT 通过设备管理推送 | 禁止访问外网 |
| 10 | System | 系统管理员 | 全局代理设置 |
| 15 | Enterprise | 企业云 bundle | 组织级模型白名单 |
| 20/21 | User | 你自己 | `~/.codex/config.toml` |
| 25 | Project | 项目仓库 | `.codex/config.toml` |
| 30 | SessionFlags | 这一次会话的 CLI 参数 | `--model o3` |
| 40-50 | Legacy | 旧版兼容 | managed_config.toml |

你写的 `~/.codex/config.toml` 是 20。项目里的 `.codex/config.toml` 是 25。25 > 20，项目赢。

## 为什么是数值，不是硬编码层数？

注意优先级是 `i16`，不是 enum 的顺序。0、10、15、20、25、30、40、50——中间有空隙。

这意味着可以在任意两层之间插入新层，不需要改已有代码。假设未来需要一个"团队级"配置（比项目大、比用户小），给它一个 22 或 23 就行。不需要重构。

角色配置（ch10 讲过的 spawn 时覆盖）就是以 SessionFlags 优先级（30）插入的。它不是"第 N 层"，它是"优先级为 30 的一个层"。

## 合并，不是选择

一个关键细节：这不是"选优先级最高的那个文件"。这是**字段级合并**。

假设 MDM 层设了 `network_policy = "whitelist"`，User 层设了 `model = "o3"`，Project 层设了 `model = "o4-mini"` 和 `approval_policy = "never"`。

有效配置是三个来源的合体：`network_policy` 来自 MDM，`model` 来自 Project（覆盖了 User），`approval_policy` 来自 Project。一个字段可能来自任何一层。它们共存于同一个有效配置中。

被禁用的层（`disabled_reason` 非空）整体跳过，不参与合并。

## AGENTS.md：另一种"配置"

AGENTS.md 不是配置文件——它是写给模型看的项目指令。但它也有自己的发现和合并规则。

从当前目录向上找，找到 `.git` 标记就是 project root。然后从 root 到当前目录，每一级的 `AGENTS.md` 都收集起来，按从根到叶的顺序拼接。

monorepo 里，根目录写"所有包必须用 cargo"，子包写"本包额外要求 nightly toolchain"。两份同时生效。根的在前，子包的在后。模型看到的是拼接后的完整文本。

还有一个 `AGENTS.override.md`——同目录下的替代文件。系统在每个目录按候选顺序查找：先找 `AGENTS.override.md`，找到就用它，不再找 `AGENTS.md`。它是替代，不是追加。适合“我个人在这个项目里的额外偏好”，不提交版本控制。

AGENTS.md 在上下文中的位置：作为 developer instructions 注入模型。它也参与 WorldState 的 baseline/diff 机制（ch06）——如果 turn 之间有人改了 AGENTS.md，只有变化部分会被记录。

## 回到开头的问题

你的 `model = "o3"` 没生效，因为项目配置优先级 25 > 用户配置优先级 20。

解决方法有三个：改项目配置（如果你是项目维护者）、用 CLI 参数 `--model o3`（优先级 30，能覆盖 Project 和 User 层）、或者接受项目的设计意图。注意：如果存在 legacy 配置（优先级 40/50），CLI 参数也盖不过它。

这不是"配置系统太复杂"。这是"配置来自多个利益相关方"的现实。公司 IT 要管控安全，项目维护者要统一行为，个人要自定义偏好，这一次会话可能有临时需求。四方诉求不同，优先级不同。两层配置（全局 + 项目）覆盖不了企业场景。

核心原则只有一条：**越具体的配置越应该覆盖越通用的配置。** MDM 适用于所有设备，所以最通用。SessionFlags 只适用于这一次会话，所以最具体。具体的覆盖通用的——这是所有配置系统设计的通用原则。Codex 只是把它显式化了。
