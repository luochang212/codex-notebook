下午两点，你开始让 Codex 重构一个认证模块。它读了 12 个文件，改了 5 个，跑了 3 次测试，修了 2 个 bug。到下午四点，你已经和它来回聊了 40 多轮。

这时候发生了一件你看不见的事：context window 快满了。

128K token 的窗口，听起来很大。但 40 轮对话里，每轮包含模型输出、工具调用、工具结果、环境信息 diff……累积下来，已经用了 110K。还剩 18K。而模型下一次响应可能就需要 5K。再跑两个工具调用，就爆了。

系统必须做点什么。但它不能简单地砍掉旧消息——那会让模型忘记自己改过哪些文件、为什么改、测试跑到哪了。

## 压缩不是截断，是换窗

Codex 的做法是：把当前全部对话历史发给模型，让它写一段摘要。然后用这段摘要**替换**整个历史。

不是截断前 N 条消息。不是只保留最近 K 轮。是让模型自己总结："我做了什么、为什么做、当前状态是什么、下一步计划是什么"。然后旧历史消失，摘要成为新的起点。

压缩后保留什么：

- 摘要文本（模型生成的）
- 文件操作记录（读过哪些、改过哪些）
- 最后一条用户消息
- 完整的 world state baseline（环境信息重新全量注入）

丢弃什么：所有中间 tool call 和 tool result、所有中间推理过程、旧的 context diff 序列。

## 压缩发生在 turn 中间，不打断你

这是最巧妙的设计：压缩不需要用户参与。它发生在 turn 循环的内部——模型调完一次工具后，系统检查 token 用量，发现超限，就地压缩，然后 `continue` 继续循环。

用户看到的是什么？什么都没有。也许模型停顿了一两秒（压缩本身是一次模型调用），然后它继续干活。它不会说"上下文快满了，我来压缩一下"。它只是继续。

## 压缩重置了 prompt cache

这里有一个隐性成本，和 ch04 直接相关。

压缩前，对话历史的前缀（base instructions + world state baseline）是稳定的，prompt cache 持续命中。压缩后，旧历史被摘要替代，baseline 重建——前缀变了。下一次请求一定 cache miss。

但只 miss 一次。从新 baseline 开始，后续请求重新积累 cache。

这是一个有意识的取舍：一次 cache miss 的成本，远低于模型因为上下文不连贯而犯错的代价。如果只截断前面而不重置 baseline，cache 前缀不变，但模型看到的上下文是断裂的——摘要和旧 baseline 对不上。Codex 选了"贵一次但连贯"，而不是"便宜但精神分裂"。

## 摘要质量是软肋

这套机制有一个无法用工程解决的弱点：摘要质量取决于模型。

如果模型写了一段很好的摘要——关键决策、文件变更、未完成任务都记下来了——后续行为几乎不受影响。但如果摘要写得差——漏掉了"用户说过不要动 auth 模块"这个约束——后续模型可能就会去动 auth 模块。

这是一个有损操作，而且损失不可预测。没有"压缩后验证摘要完整性"的机制。压缩完就完了，模型基于摘要继续工作。

## 两种压缩路径

Codex 支持本地压缩和远程压缩：

- **本地**：用当前模型做摘要，消耗本地 token budget
- **远程**：provider 支持时，压缩在服务端完成，不算在用户的 token 用量里

远程压缩的优势是成本透明——用户不会因为"系统自己做了一次摘要"而被多收钱。但不是所有 provider 都支持。

## 回到那个下午四点

你的 session 在压缩后继续运行。模型看到的是一段几百 token 的摘要，而不是 110K token 的完整历史。它知道自己改过哪些文件、测试状态如何、下一步要做什么。但它不再记得第 7 轮时你说过的一句随口评论，也不再记得第 15 轮时它尝试过但放弃的一个方案。

这是有意的。那些细节在当时的上下文里有意义，但对"接下来做什么"已经不重要了。如果它们真的重要，摘要会记住。如果不重要，忘了就忘了。

对话可以无限延续。代价是：越远的记忆越模糊。就像人一样。
