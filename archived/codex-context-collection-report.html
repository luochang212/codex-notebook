Codex 如何为一次 Query 组织上下文？

它不是先从旧对话里检索“最相关片段”，而是维护连续会话历史，再叠加运行时状态差量、显式能力注入与按需工具结果；窗口吃紧时，才用有损摘要接力。

Codex Core 负责组织和约束上下文，不负责对全部历史做通用语义相关性排序。旧信息是否与当前 query 有关，主要由模型在读取上下文后判断。

Request lifecycle

一次 query 不是单独发送，而是被放回整个活动上下文

run_turn 是正常 turn 的主入口。它先判断是否需要压缩，再捕获本轮可见的运行状态，处理 skill、plugin、hook 和用户输入。真正准备模型输入时，它复制当前历史并调用 for_prompt。

01 / CHECK检查窗口预算

达到阈值时，先执行 pre-turn compaction。

02 / CAPTURE捕获当前状态

固定本次请求所见的环境、工具和执行配置。

03 / RECORD追加本轮内容

记录用户输入、hook 上下文和显式能力注入。

04 / SAMPLE整理完整历史

规范化活动 history 后，将其交给模型。

codex-rs/core/src/session/turn.rs::run_turn关键证据
// Construct the input that we will send to the model.
let sampling_request_input = sess
    .clone_history()
    .await
    .for_prompt(&turn_context.model_info.input_modalities);

注意这里没有传入当前 query，也没有出现 search_relevant_history(query) 一类调用。概念上，模型收到的是当前活动历史，而不是 Core 根据 query 预选出的 top-k 对话片段。

Continuity

窗口内连续，但不是无限、逐字、无损

ContextManager 使用按时间顺序排列的 Vec<ResponseItem> 保存 thread transcript。用户消息、assistant 回复、reasoning、工具调用和工具输出都会进入这个序列。

CONTINUOUS

顺序追加

旧 item 在前，新 item 在后。同一 turn 内的工具输出写回历史后，会参与下一次 sampling。

LOSSY EDGES

存在有损边界

工具输出可能被截断；不支持的媒体会被剥离；compaction、rollback 和新窗口会改写历史。

for_prompt 主要负责规范化：补齐工具调用/输出配对、删除孤立输出，并根据模型支持的输入模态剥离图片或音频。它不是语义过滤器。

Baseline + diff

对话文本靠追加，运行时状态靠基线与差量

Codex 不会每轮都重复塞入完整环境。它将 cwd、workspace roots、权限、日期时区、协作模式、AGENTS.md、apps 与 plugins 等组织为 WorldState。

codex-rs/core/src/session/mod.rsfull vs diff
let should_inject_full_context = reference_context_item.is_none();

if should_inject_full_context {
    // 没有可信基线：完整注入
    build_initial_context_with_world_state_and_mcp(...)
} else {
    // 稳态：只追加 settings 与 world-state 的变化
    history.update_world_state(world_state)
}

这里解决的是重复和状态一致性，不是语义相关性。如果 baseline 消失，系统会重新全量注入，避免在缺少前态时继续做 diff，导致模型看到不完整状态。

关键区别：状态 diff 回答“什么发生了变化”；它不回答“什么与当前问题语义最相关”。

Context checkpoint

上下文过长时，连续历史会被压缩为一份任务交接

达到自动压缩阈值或模型完整上下文窗口时，Codex 让模型生成 handoff summary，要求保留进展、关键决策、约束、用户偏好、剩余工作和必要引用。压缩完成后，它会替换活动历史，而不是简单在原历史末尾追加摘要。

本地 compaction 会从新到旧选择真实用户消息，合计最多约 20,000 token，再追加摘要。必要时，当前 initial context 会被重新插入到最后一个真实用户消息之前。

因此压缩后的连续性是“语义接力”，不是逐项回放。多次压缩可能累积遗漏和误差，源码也明确提醒长 thread 与多次 compaction 会降低准确性。

Relevance boundaries

“相关”在 Codex 中有四种不同含义

上下文类型如何进入请求谁判断相关性
旧对话活动窗口内按顺序整体保留主要由模型根据当前 query 判断
运行时状态按当前 cwd、环境与配置确定作用域，再做 diff确定性作用域规则，不做语义打分
Skill / plugin / app检查当前用户输入中的显式 mention，并注入对应指导结构化路由与显式触发规则
仓库与外部材料模型调用搜索、文件、Git、MCP 等工具后写回 history模型决定搜什么、读什么

历史消息：没有默认的 top-k 语义检索

如果早期讨论任务 A，之后转向无关的任务 B，A 仍会留在活动窗口中。Core 不会因为它和 B 不相似就立即删除它。最可靠的隔离方式仍然是为独立任务创建新 thread。

能力注入：少数真正检查当前 query 的路径

build_skills_and_plugins 只从当前 TurnInput::UserInput 收集内容，用于识别显式 skill、plugin 和 app mention。它是规则驱动的 turn-specific routing，但并不是对全部会话历史做通用语义检索。

代码材料：相关性形成于工具闭环

Codex 不会在每次 query 前自动扫描整个仓库。模型先看到 query、历史、运行环境和工具定义，再决定搜索与读取哪些内容。工具结果进入 history，模型据此继续推理。

Optional memory layer

长期 memory 是另一层，不等于当前 thread history

启用 MemoryTool 且配置允许时，memories extension 会把 memory_summary.md 作为 developer context 注入，并提供进一步读取或搜索 memory 的工具。

这仍不是“每轮自动对所有长期记忆做 embedding top-k”：摘要是预先整理的高密度入口，更详细内容需要模型按需使用工具取得。该层面向跨 thread 经验，而 ContextManager 维护的是当前 thread 的活动上下文。

Working mental model

把 Codex 上下文理解为四层叠加

LAYER 1

基础指令

模型、开发者和协作模式指令，构成相对稳定的约束前缀。

LAYER 2

连续历史

用户、assistant、reasoning 和工具事件，按时间顺序追加。

LAYER 3

运行时状态

首轮全量，后续差量；按执行作用域确定，而非语义相似度。

LAYER 4

按需材料

文件、Git、网页、MCP 与 memory，通过工具调用进入历史。

Codex 的上下文在活动窗口内基本连续累积，但不会在每轮开始时智能删除所有“与当前 query 无关”的旧消息。它用作用域规则、显式 mention、状态 diff、按需工具和 compaction 控制上下文；真正的语义相关性判断主要留给模型。
